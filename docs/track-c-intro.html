Foundation

What is an AI agent?

A language model that doesn't just answer — it decides what to do and does it.

Perceive

Reads files, code, data, PDFs, terminal output, web pages.

Plan

Decides which tools to use, in what order, before acting.

Act

Writes files, runs code, calls APIs. Output lives on disk.

Remember

Context window this session · CLAUDE.md every session.

Iterate

Observe → Plan → Act → Observe → … until done.

Chatbot: text in, text out.  Agent: goal in, result on disk — or in a web form, or in your lab system.

Crash Course — Tool Choices

Two tools. Different jobs.

ChatGPT

Text in. Text out.

  • Replies once, then waits for you
  • No access to your files or disk
  • No memory between sessions
  • You copy-paste everything manually
  • Good for: quick questions, drafting, brainstorming
Claude Code (AI Agent)

Goal in. Result on disk.

  • Reads your files — code, data, PDFs, notebooks
  • Runs terminal commands — python, bash, git
  • Saves output to disk — scripts, reports, figures
  • Loops until the task is done — plan → act → fix → repeat
  • Persists instructions across sessions via CLAUDE.md
The shift: Claude Code doesn't produce text for you to copy — it produces files on your disk, code that runs, web pages that open, forms that fill. Today we use Claude Code.

Foundation

Start with a folder. Tell Claude who you are.

Claude Code's world is a directory. Start there — navigate to your project folder, type claude, run /init.

CLAUDE.md — your standing instructions

Plain text file at the root. Claude reads it automatically at the start of every session — without you saying anything. Keep it to 10–20 lines.

What to put in it

Project purpose, data formats, file layout, conventions ("never modify raw data"), constraints. Generate it instantly with /init, then edit.

Git = undo for everything

Run git init before your first session. If Claude does something unexpected, git restore . brings it all back.

Example CLAUDE.md for a KTH admin agent
# KTH Scheduling Agent Project: automate recurring scheduling emails and meeting coordination for our group. Tools available: Python, browser (Claude Chrome), KTH Outlook calendar via web UI. Output: draft emails to ./drafts/, logs to ./logs/. Never send emails without my explicit approval. Language: English for external, Swedish for internal. Contacts file: ./data/contacts.csv columns: name, email, role, timezone
Keep CLAUDE.md short. It loads every session and burns tokens every time. Use Skills (slide 7) for knowledge that only applies to specific tasks.

The Key Mindset Shift

The question is not: what prompt should I write?
The question is: what should Claude know?

Context engineering — deliberately shaping what's in the context window — is the core skill.

The Constraint

Context fills fast

Every file read, every command output, every message enters the window. Performance degrades as context fills — Claude starts forgetting earlier instructions.

10 files + 20 messages = context half gone before the real task starts.
Three Levers

Shape what Claude knows

CLAUDE.md — always loaded; @file — read when you mention it; Skills — description pre-loaded (~100 tokens), full body only on trigger.

CLAUDE.md for universal rules · Skills for domain workflows · @file for one-off context.
Rich Context → Rich Output

The compounding effect

Vague prompt + blank context = generic output. Precise task + tight CLAUDE.md + right files = working domain-specific results.

"Fill the eISP form" → fumbles. "Read student-data.csv, open eISP at the URL below, fill these fields" → done.
Context Management

Keep it clean

/clear — reset between unrelated tasks. /btw — side question that never enters history. Subagents — investigate in a separate window.

Corrected Claude twice on the same issue? /clear and restart with a better prompt.

Daily Usage

The commands you need today.

Start & Run
claude
Start interactive session — reads directory + CLAUDE.md immediately
claude -p "task"
One-shot non-interactive run — pipe into scripts or batch automation
--permission-mode auto
Auto-approve routine actions; only prompts for risky steps
--dangerously-skip-permissions
Advanced: fully autonomous — skips all prompts. Use only in git-tracked projects where you can revert.
Build Context
/init
Generate CLAUDE.md from current project — run once, then edit
#add a convention
Append a line to CLAUDE.md without executing — build incrementally
Manage Context
/clear
Reset context — use between unrelated tasks
/btw [question]
Side question that never enters conversation history
Control & Undo
Esc · Esc+Esc
Interrupt running action · restore previous file state

Extending Claude Code · Skills

Skills: domain knowledge that costs ~100 tokens.

Progressive disclosure — zero cost until triggered

At startup: only name + description loaded (~100 tokens total for all skills). When triggered: full SKILL.md body loaded. Reference files: loaded only when needed.

CLAUDE.md vs Skills — when to use which

CLAUDE.md — loaded every session, always burns tokens. Use for rules that apply to every task.
Skills — triggered on demand. Use for domain workflows, system schemas, step-by-step procedures.

Install: one folder, one file

Create .claude/skills/my-skill/SKILL.md — that's it. Claude discovers it automatically. Find community skills at github.com/modelcontextprotocol/servers

The description is critical

Claude picks the right Skill using only the name + description. Write: what it does + when to use it. Vague descriptions → Skill never triggers.

Minimal SKILL.md for a KTH system
--- name: kth-eisp-form description: Fill KTH eISP individual study plan forms for PhD students. Use when asked to update, submit, or review eISP documents, or when working with PhD student plans. --- ## eISP workflow 1. Open eISP at https://eisp.sys.kth.se 2. Search student by KTH-id 3. Fill fields per ./data/student-template.csv 4. Save as draft — NEVER submit without approval ## Field mapping - supervisor_1: primary supervisor (required) - half_time: expected half-time date (YYYY-MM-DD) - courses_planned: comma-separated course codes ## See also [FIELD-REFERENCE.md](FIELD-REFERENCE.md) — all 24 fields [SUBMIT-CHECKLIST.md](SUBMIT-CHECKLIST.md) — pre-submit review

Extending Claude Code · Build Your Own Skill

Write a Skill. Capture what you know.

The "grill with docs" pattern

One of the most useful Skill patterns: interview-style questioning about a plan or design, one question at a time, with recommended answers. Explores the codebase or docs before asking — never relies solely on user input.

How to build a Skill in practice

① Complete a task without a Skill — note what context you kept providing repeatedly.
② Ask Claude to write a SKILL.md that captures that reusable pattern.
③ Test with a fresh session. Iterate based on what Claude misses or gets wrong.

Skill file structure

Keep SKILL.md under 500 lines. Longer reference material → separate files. Claude reads them on demand via progressive disclosure — zero token cost until needed.

The description must say when to trigger

Bad: "Helps with documents."   Good: "Processes KTH Canvas course exports and generates structured learning outcome reports. Use when working with Canvas data or course analytics."

grill-with-docs pattern (adapted)
--- name: grill-with-docs description: Stress-test a plan against existing docs, schemas, or code. Use when user wants to review a design, validate assumptions, or clarify terminology against existing material. --- ## Process 1. EXPLORE first — read relevant files/docs before asking any questions 2. Ask ONE question at a time 3. Provide a recommended answer for each question 4. Challenge terminology against known schemas 5. Cross-reference claims against actual code ## Question types - Terminology: "Does 'user' here mean KTH-id holder or anyone with a local account?" - Edge cases: "What happens when the API is down?" - Dependencies: "Which service owns this data?" ## After grilling Update docs/CONTEXT.md with resolved terms. Create an ADR only for hard-to-reverse decisions.
Source: github.com/mattpocock/skills (MIT)

Extending Claude Code · External World

CLI tools and browser automation.

CLI Tools
  • Claude uses gh, curl, aws, gcloud natively — already authenticated, no setup
  • New tool with a CLI? Just say: "Use foo --help to learn it, then do X"
  • Scripts are better than MCP for most research tasks — zero server overhead
  • KTH: canvas-cli, ladok-api wrappers if available — or curl + KTH API token
Claude for Chrome
  • Built into Claude Desktop Cowork mode
  • Reads web pages, fills forms, navigates, clicks
  • Best for: KTH web UIs (eISP, Canvas, ordering systems) where no API exists
  • Easiest to get started — already installed if you set up Cowork
  • Grant tab access per site when prompted
Playwright MCP / agent-browser
  • Playwright MCP: full browser control via Model Context Protocol
  • agent-browser (Vercel Labs): headless, Playwright-based, designed for agent tasks
  • Best for: complex multi-step workflows, screenshot capture, structured data extraction
  • More setup than Chrome extension but more powerful and scriptable
  • github.com/vercel-labs/agent-browser
Rule of thumb: if a CLI exists → use CLI. If you need a web UI and Cowork is installed → Claude for Chrome. If you need headless batch scraping or complex multi-page automation → Playwright MCP or agent-browser.

Hands-On · Start Here

Four steps to your first working version.

One person sets up the repo. Others clone and join. Start together.
Step 1

Create a Git repo

mkdir my-project
cd my-project
git init

Push to GitHub and add teammates as collaborators. Others: git clone <url>

Step 2

Start Claude Code

claude

Claude reads the folder immediately. Run /init to generate a CLAUDE.md. Edit it to describe your project in 5–10 lines.

Step 3

Tell Claude what you want

Describe your goal in plain language — outcomes, not steps. Mention relevant files by name. "Read contacts.csv, draft scheduling emails for next week's meetings."

Step 4

Iterate toward the demo

Small follow-ups compound. "Handle the case where the email field is empty." "Add a confirmation step." Keep refining the same thread rather than restarting.

If you get stuck: say "read the error and fix it." Don't copy-paste anything yourself — that's the agent loop. If you're stuck on direction, grab Wei.
