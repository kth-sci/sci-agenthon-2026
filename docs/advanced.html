1. Three Mental Models for the New Era

Software 1.0 → 2.0 → 3.0

Karpathy frames the current moment as a third software paradigm. In Software 1.0, humans write explicit rules as code. In Software 2.0, humans curate datasets and train neural networks — the program is learned into the weights. In Software 3.0, humans write prompts. The LLM is the interpreter; the context window is your primary lever.

The shift in mindset: What is the piece of text to copy-paste to your agent? That's the programming paradigm now. The agent has its own intelligence — you don't have to precisely spell out every detail. It looks at your environment and follows intelligent instructions.

Software 1.0

Humans write explicit rules in code. Works well for things you can fully specify. Deterministic.

Software 2.0

Humans curate datasets and objectives. Neural networks learn the program. Training = programming.

Software 3.0

Humans write prompts, context, Skills, examples, memory. The LLM interprets. Context is code.

Jagged Intelligence & Verifiability

LLMs have jagged intelligence — they spike in capability wherever tasks are verifiable (testable, rewardable), heavily trained by labs, and economically valuable enough for labs to prioritize. They stagnate — sometimes bizarrely — outside those circuits.

How is it possible that Opus 4.7 will simultaneously refactor a 100,000-line codebase or find zero-day vulnerabilities, and yet tells me to walk to the car wash 50 meters away? This is insane. It means you need to actually be in the loop and treat these as tools.

— Andrej Karpathy, Sequoia Ascent 2026

The practical question to ask constantly: "Am I on the model's rails?" — are you operating in a verifiable, heavily-trained domain (code, math, structured text) or drifting into circuits where performance is unpredictable?

LLMs as Operating Systems

Karpathy argues LLMs are better understood as a new kind of operating system than as a utility like electricity. The analogy: context window = working memory, LLM = CPU orchestrating resources, tool use = expanding OS capabilities. Closed-source providers parallel Windows/macOS; open-source models parallel Linux.

Practical implication: just as software had to be rewritten when operating systems changed, all infrastructure must now be redesigned for agents as primary users — docs, APIs, logging, permissioning, setup flows.

LLM Psychology: What You Must Never Forget

From Karpathy's "Software is Changing Again" talk — a practical mental model for working with LLMs without being caught off-guard:

Superpowers

Encyclopedic memory surpassing individual humans. Pattern recognition across domains. Knowledge synthesis from internet-scale training. Remarkably good at verifiable, heavily-trained tasks.

Cognitive deficits

Hallucination — fabricates information confidently. Jagged intelligence — superhuman in some domains, comically wrong in others. Anterograde amnesia — cannot learn from interactions; context is wiped between sessions. Gullibility — susceptible to prompt injection.

Memento and 50 First Dates — protagonists with fixed memories and wiped context each morning. That's the LLM condition. It's a useful mental model for why you need CLAUDE.md, why you need Skills, and why you must stay in the loop.

— Karpathy, "Software is Changing Again" (via Memento analogy)
Partial autonomy beats full autonomy. Karpathy's lesson from self-driving cars: "I get concerned seeing '2025 is the year of agents.' This is the decade of agents." Build products that put agents on an autonomy slider — from tab completion to full autonomy — and let users control delegation. Resist flashy demos of full autonomy; build the fast human-AI verification loop instead.

Animals vs. Ghosts

Karpathy argues we are not building animals — LLMs lack biological drives, embodiment, or intrinsic motivation. We are summoning ghosts: statistical simulations shaped by pretraining, post-training RL, and economic incentives. Practical implications:

  • Don't anthropomorphize. Yelling at them doesn't make them work better or worse.
  • Treat as empirical tools. Learn where they reliably work and where they don't in your domain.
  • They are artifacts of training choices — their capabilities reflect what labs prioritized and rewarded.
  • Neither dismiss nor blindly trust. Stay in the loop on anything critical.

2. Context is Your Lever — But It Degrades

Context engineering — deliberately shaping what's in the context window — is the core skill of working with AI agents. But research shows performance grows increasingly unreliable as input length grows, even on intentionally simple tasks. This is "context rot."

What Context Rot Does

  • Performance degrades non-uniformly — some distractors are far more disruptive than others
  • Models under-generate, hallucinate, or refuse tasks they'd handle easily in short contexts
  • Structured, coherent haystacks can hurt performance vs. shuffled content (attention anomaly)

What Helps

  • Keep contexts short and focused. Start new sessions for unrelated tasks.
  • Use /clear aggressively — don't let unrelated history accumulate
  • Progressive disclosure: only load what's needed, when it's needed (Skills pattern)
  • Put critical information right before where you need it — recency matters
  • Claude models show lowest hallucination rates and lean toward abstention when uncertain

Context Engineering vs. Prompt Engineering

These are not the same thing. Prompt engineering optimises the instruction sent to a model — how you ask. Context engineering curates the entire information architecture during inference: code, decisions, documentation, tool results, conversation history, examples — what the model knows when it answers.

You can write a brilliant prompt, but if it's buried behind thousands of tokens of irrelevant chat history or poorly structured retrieved documents, the model won't follow it. Context engineering builds the container that gives the prompt room to work.

— Port.io, "Context Engineering vs Prompt Engineering" (2025) · port.io/blog

Prompt engineering gets you the first good output. Context engineering makes sure the 1,000th output is still good. Endorsed publicly by Andrej Karpathy and Shopify CEO Tobi Lütke in mid-2025 — Shopify now requires teams to demonstrate AI cannot do a job before requesting human headcount.

The Three Levers

CLAUDE.md — always loaded

Loaded at the start of every session, consuming tokens every time. Use only for rules that apply to every task. Keep ruthlessly short (10–20 lines). Every token here is always spent.

@file — on demand

Mention a file by name and Claude reads it. Zero cost until you reference it. Use for one-off context that only applies to the current task.

Skills — triggered by description

Only the name + description (~100 tokens total for all installed skills) is pre-loaded. Full body loads only when the skill is triggered. Reference files load only when needed. Zero cost until relevant — the most token-efficient way to give Claude deep domain knowledge.

The question is not: what prompt should I write? The question is: what should Claude know?

— Claude Code documentation

The 35-Minute Wall and Lost-in-the-Middle

Two empirical findings that should shape how you run long Claude Code sessions:

The 35-minute wall (coding agents)

Beyond 35 minutes of continuous agent work, doubling task duration quadruples the failure rate. This is the point where context accumulation crosses 80K–150K tokens. Agents spend over 60% of their first turn just retrieving context, not reasoning. Break long sessions into checkpointed sub-tasks.

Lost-in-the-middle effect

Liu et al. (2024, TACL) found that accuracy drops significantly when relevant documents appear in the middle of a long context vs. at the start or end — attention is U-shaped. Put critical instructions early (before context is half full) or at the very end. Never let key guidance drift to mid-context as history accumulates.

Coherent documents are worse distractors. Counterintuitively, well-structured, logically coherent haystacks create more confusion than shuffled ones — structural coherence makes irrelevant passages look more plausible to the attention mechanism. This is why RAG chunking strategies matter more than they look.

3. Agent Design Patterns

Anthropic's canonical guide distinguishes workflows (fixed code paths) from agents (LLM-directed). Start with the simplest pattern that works.

Start simple first. Optimized single LLM calls, retrieval augmentation, and in-context examples often outperform elaborate agentic systems. Add complexity only when simpler approaches demonstrably underperform. Agentic systems impose latency and cost for every hop.

① Prompt Chaining

Sequential LLM calls where each processes the previous output. Best for: fixed, decomposable subtasks where accuracy gains justify latency. Example: draft outline → validate → write full document.

② Routing

Input classification directing to specialized downstream processes. Best for: distinct query types needing different handling. Example: route to Haiku for simple questions, Sonnet for complex reasoning.

③ Parallelization

Independent subtasks run simultaneously, or same task run multiple times for consensus. Best for: speed improvements or confidence through multiple perspectives. Example: 3 agents review code for vulnerabilities, flag issues if ≥2 agree.

④ Orchestrator–Workers

Central LLM dynamically decomposes tasks, delegates to specialist workers, synthesises results. Best for: complex tasks where subtask structure varies per input (unlike preset parallelization). Example: multi-file code changes where affected files depend on the task.

⑤ Evaluator–Optimizer

Generator LLM produces output; evaluator LLM provides feedback in iterative loops. Best for: tasks with clear evaluation criteria where iterative improvement measurably helps. Example: translation → critique → refine → repeat until quality threshold met.

Tool Design: The Agent-Computer Interface (ACI)

Anthropic found that in building their SWE-bench coding agent, they spent more time optimizing tools than optimizing the overall prompt. The lesson: tool design is as important as prompt design.

Use natural formats

Match formats that naturally occur in internet text. Avoid unusual structures, unnecessary escaping, or formats the model would rarely have seen during training.

Document like you're hiring a junior dev

Include usage examples, edge cases, input format specs, and boundaries. Names should be obvious. Docstrings should be comprehensive.

Poka-yoke your tools

Redesign tool arguments to prevent common errors rather than handling them after. Example: require absolute paths to eliminate relative path confusion.

Test with varied inputs

Identify failure patterns through workbench testing. Every tool should be stress-tested the same way you'd test code for human users.

4. Skills over MCP: Why and When

MCP (Model Context Protocol) was positioned as the universal standard for agent tool integration. In practice, it has a serious context cost problem — and for most research and admin workflows, a combination of CLIs and Skills is far more efficient.

The MCP context cost [Zhang, 2025]: Three MCP servers (GitHub, Playwright, IDE integration) consumed 143K tokens on a 200K-token model, leaving less than 30% usable space. The author observed tool selection accuracy drop from 43% to below 14% as tool count grew — every tool definition loads into context even when unused, compounding context rot.

Use CLI tools

For anything with a good CLI — gh, curl, aws, gcloud. Claude uses these natively. Already authenticated, no server process, zero context overhead until called. New tool? "Use foo --help to learn it, then do X."

Use Skills

For domain knowledge, workflows, and system procedures. Description pre-loads (~100 tokens), body loads only on trigger, reference files load only when needed. Actively guides workflows rather than passively waiting to be called.

Use MCP when

You need real-time bidirectional app control (Blender, Figma, live databases). Browser automation via Playwright MCP. Anything requiring two-way state sync that a CLI can't do. Enterprise governance scenarios.

MCP's architectural problems

  • Flat tool design: MCP treats all tools as flat function signatures — no hierarchy, no subcommands, no workflow awareness. CLIs naturally express git commit vs git push; MCP can't.
  • No LLM reuse: MCP servers can't access the agent's primary language model for filtering or ranking results.
  • Passive tools: MCP tools wait to be called. Skills actively guide — they contain the procedure, not just the capability.
  • Setup overhead: Requires a running server process + config file + connection setup. Skills are just a folder with a markdown file.

For most research workflows: MCP is overkill. The better default is Skills + CLI tools. Same power, a fraction of the token cost.

— Claude Code documentation

5. The Human Role: What You Can't Delegate

You can outsource your thinking, but you can't outsource your understanding.

— Tweet cited by Karpathy as "blew my mind, I keep thinking about it every other day"

As agents take over execution, human understanding becomes the bottleneck, not human coding speed. You can delegate syntax recall and implementation details. You cannot delegate knowing what to build, why it matters, or whether the output is subtly wrong.

What becomes more valuable

Taste and judgment

Determining what's worth building, which questions matter, identifying suspicious results. Current agents produce working code but often bloated, copy-pasted, awkwardly abstracted, brittle. Taste is not rewarded in RL — it's still human territory.

System design knowledge

Storage, memory copies, invariants, identity, security boundaries. The MenuGen payment bug: agent matched Stripe emails to Google emails — plausible but wrong. The fix (use persistent user IDs) requires system design intuition agents don't have.

Specification and planning

Working with agents to design detailed specs before implementation. The agent fills in technical details; you own the top-level categories and their correctness. "You're in charge of the spec, this plan."

Domain-specific correction signals

When output is wrong in a domain-specific way, you must explain why it's wrong. "In this system, you must never match across email providers — use the user_id foreign key." That context doesn't exist in the model's training.

What becomes less scarce

Code generation API recall Boilerplate First drafts Repetitive setup Simple transformations Syntax recall Literature search
Karpathy on code quality: "When I look at the code, sometimes I get a little bit of a heart attack. It's not like super amazing code necessarily all the time — it's very bloaty, there's a lot of copy-paste, there are awkward abstractions that are brittle, and it works but it's just really gross. Aesthetics and simplicity are not part of the RL reward — that's still on you."

6. Karpathy's Four Rules for Claude Code

From the karpathy-guidelines SKILL.md — a codified set of behavioral rules designed to reduce the most common LLM coding mistakes. "Bias toward caution over speed."

① Think Before Coding

"State your assumptions explicitly. If uncertain, ask." Surface tradeoffs rather than making silent decisions. Clarify confusion before implementation, not after. Never guess at a constraint when you can ask.

② Simplicity First

"Minimum code that solves the problem. Nothing speculative." Avoid unrequested features, unnecessary abstractions, or speculative error handling. Test: would a senior engineer call this overcomplicated? If yes, strip it back.

③ Surgical Changes

"Touch only what you must. Clean up only your own mess." When editing code, preserve existing style. Avoid improving unrelated areas even if you see issues. Remove only the dead code your changes created.

④ Goal-Driven Execution

"Define success criteria. Loop until verified." Transform tasks into testable objectives with clear verification steps before starting implementation. Don't finish until the goal is demonstrably met.

7. Matt Pocock's Skill Library

21 opinionated dev workflows refined from real production TypeScript work. The key idea: encode discipline into tooling, not into your memory. When the tdd skill is active, Claude cannot write implementation before tests — the constraint is in the skill, not in your prompt.

tdd — Test-driven development

Red-green-refactor loop enforced at the skill level. Builds features or fixes bugs one vertical slice at a time. Claude cannot skip the test step when this skill is loaded.

grill-me — Adversarial design review

Turns Claude into a relentless technical interviewer. Explores the codebase first, then asks one question at a time with a recommended answer. Hunts every branch, edge case, and assumption. Forces you to defend your design.

grill-with-docs — Design vs. documentation

Stress-tests a plan against existing docs, schemas, or code. Challenges terminology, tests relationships via edge cases, cross-references claims against actual code. Updates CONTEXT.md as terms resolve. Creates ADRs only for hard-to-reverse decisions.

to-prd — Context to GitHub issue

Turns the current conversation into a Product Requirements Document and submits it as a GitHub issue. Captures what you've discussed without you having to re-explain it to a human.

git-guardrails — Safe git operations

Certain git operations require explicit human approval when this skill is loaded. Encodes caution for destructive operations into the tool rather than relying on the prompt.

ubiquitous-language — Terminology consistency

Maintains a shared glossary (CONTEXT.md) as you build. When Claude names a concept, it must match or justify deviating from existing terms. Kills terminology drift in long projects.

Pocock's philosophy: AI is an opinionated team member who has internalized your workflow standards. Constraints are encoded in the tooling, not dependent on you remembering to include them in every prompt.

— Distilled from mattpocock/skills README and documentation

grill-with-docs: Full Workflow

The grill-with-docs skill merges two patterns: grill-me (relentless questioning) and ubiquitous-language (shared glossary). It stress-tests a design against existing docs, schemas, or code — one question at a time, with recommended answers. The agent explores the codebase before asking, so questions are grounded in what actually exists.

① CONTEXT.md as foundation

The skill searches for a CONTEXT.md file containing the shared glossary for this bounded context. This is a pure glossary — totally devoid of implementation details. It's the AI's knowledge base for the project's domain language. Multi-context repos include a CONTEXT-MAP.md pointing to domain-specific folders.

② One question at a time, with a recommended answer

The AI asks cardinality questions ("is this 1:1 or 1:N?"), terminology collision questions ("this term conflicts with X already in CONTEXT.md"), status semantics, deletion behavior, edge case scenarios. For each, it gives a recommended answer — you can accept or override. After each answer, it updates CONTEXT.md inline.

③ Verify against the actual codebase

When implementation exists, the agent reads the code to check whether behavior matches claims. It surfaces contradictions immediately rather than relying on your description. This is what makes it different from a generic design review — it reads first, asks second.

④ ADRs only for hard decisions

Create Architectural Decision Records only when a decision is hard to reverse, surprising without context, and involves genuine trade-offs. "Should this be ON DELETE CASCADE or RESTRICT?" is ADR territory. "Should I name this variable 'user' or 'account'?" is CONTEXT.md territory.

Why it works beyond the immediate session: "With shared language documented, the AI uses fewer tokens — instead of verbosely re-describing everything, it operates efficiently." Code navigability improves because planning documents align with how the code actually looks. Developers, domain experts, and the codebase all speak the same language.

Install

npx skills@latest add mattpocock/skills/skills/engineering/grill-with-docs

Or read the full SKILL.md at github.com/mattpocock/skills

How to write effective Skills

From Anthropic's skill authoring best practices:

The description is everything

Claude picks the right skill using only the name + description. Write: what it does + when to use it. "Processes KTH Canvas course exports and generates structured learning outcome reports. Use when working with Canvas data or course analytics." — not "Helps with documents."

Progressive disclosure

SKILL.md is a table of contents, not an encyclopedia. Keep the body under 500 lines. Move longer reference material to separate files (REFERENCE.md, FIELD-MAP.md) and point to them. Claude reads them on demand — zero token cost until needed.

Build it with Claude, then test it with Claude

Use "Claude A" to create the skill by capturing what you repeatedly explained during a task. Then test it with "Claude B" (fresh session). Watch where B struggles or misses — bring that back to A. Iterate from observed behavior, not assumptions.

Concise is key

Only add context Claude doesn't already have. Challenge every line: "Does Claude really need this explanation? Can I assume Claude knows this?" A concise skill that triggers reliably beats a verbose one that Claude partially reads.

8. The LLM Wiki Pattern

Karpathy's most underrated idea: instead of RAG answering questions from raw documents, use agents to incrementally compile documents into a structured Markdown wiki — with summaries, entity pages, concept pages, cross-links, and evolving synthesis.

No classical program could robustly maintain that kind of knowledge base across messy human documents. But an LLM can. Every time I read an article I have my wiki that's being built up. I love asking questions about things. These are tools to enhance understanding, not just answer machines.

— Andrej Karpathy, Sequoia Ascent 2026

Why Markdown?

Portable forever

Opens anywhere. Lock-in free. Sync via git, use any editor, maintain full ownership. Still readable in 20 years.

Native to LLMs

Models understand Markdown structure natively. Headers, lists, and links are meaningful to the model, not just formatting.

Clarity-forcing

The format nudges toward well-organized information. Bad notes become obvious; good notes become queryable.

Building your wiki

Consistent structure per note

Title, one-line summary, tags, timestamps, content. The summary line is what Claude uses to decide if this file is relevant — it's the per-file "description".

Write summaries, use consistent terminology

Pick one term per concept and use it everywhere. [[wiki links]] create richer reasoning graphs. Focused notes outperform sprawling ones.

Query via Claude Code

Navigate to the wiki folder and type claude. Ask: "What notes do I have about X?" or "Summarise what we know about Y across all files." Direct file-reading works for hundreds of notes; add RAG (LlamaIndex) for thousands.

For Track C: This pattern is directly applicable — build a knowledge agent over your group's papers, internal reports, or KTH documentation. The agent incrementally compiles a wiki; you query it in natural language. No classical search, no RAG infrastructure needed for the prototype.

9. Designing Agent-Native Systems

Everything is still fundamentally written for humans. Docs tell you what to do. GUIs require you to click. This is Karpathy's "favourite pet peeve": "I don't want to do anything. What is the thing I should copy-paste to my agent?"

The agentic era requires rewriting infrastructure with agents as primary users. The framework: sensors convert world state to digital information that agents can read; actuators let agents change things.

What agents need (design for this)

  • Markdown documentation (not just HTML tutorials)
  • CLIs and APIs (not just GUIs)
  • Structured, machine-readable logs
  • Defined schemas with explicit field names
  • Copy-pasteable agent instructions (not step-by-step for humans)
  • Safe permissioning models with clear capability boundaries
  • Auditable action trails
  • Headless setup flows

The question to ask

Instead of: "What existing workflows can AI speed up?"

Ask:

  • What information transformations were impossible before but are now natural?
  • What software shouldn't exist as an app — and should just be a model call?
  • What verifiable domains exist that labs haven't heavily trained on yet?
  • Where can we create our own domain-specific feedback loops?

The MenuGen web app shouldn't exist. The Software 3.0 version is: take a photo, give it to a model, say "overlay dish images onto the menu." The neural network does the work, no app needed in between. Some apps are working in the old paradigm.

— Andrej Karpathy, Sequoia Ascent 2026

Simon Willison: Practical wisdom

Speed enables ambition

The primary benefit of AI agents isn't faster coding — it's being able to ship projects previously considered too time-consuming to justify. This creates a compounding loop: faster execution → more experimentation → deeper expertise.

Be ready to take over

Sometimes manual intervention beats continued prompting. If you've been fighting the LLM on something for 10 minutes, switch to the docs and fix it yourself. Pragmatism over principle.

Favour "boring technology"

Stable, well-documented libraries that were heavily present in pretraining data outperform bleeding-edge tools the model barely knows. The model's knowledge cutoff is real — provide recent docs for anything that changed after it.

10. Trust Escalation & Sandboxing

The field is moving in one consistent direction: give agents more autonomy, protect that autonomy with hard system boundaries rather than permission prompts. The design question is shifting from "what tools should the agent have?" to "what system-level boundaries keep the agent safe to run freely?"

Fewer tools, better agents

Vercel's v0 team built a text-to-SQL agent with 15+ specialised tools — schema lookup, query validation, error recovery, intent clarification, join-path finding. Then they removed 80% of them, leaving only bash and SQL execution. Results [Vercel, 2025]:

We were doing the model's thinking for it. Each tool was a constraint we built because we didn't trust the model to reason. With Claude Opus 4.5, those guardrails became counterproductive.

— Vercel engineering team, "We removed 80% of our agent's tools" (2025)
The prerequisite: this only works when you have excellent underlying data/documentation. Vercel's semantic layer had well-organized YAML with clear definitions and consistent naming — the agent could reason freely because the foundation was clean. Garbage-in still means garbage-out.

The new design principle

Start with model + filesystem + goal

The minimal viable agent: one capable model, access to a filesystem it can read and write, and a clear objective. Add tools only when the agent demonstrably cannot do something it needs to do. This is the opposite of the old approach (enumerate every capability upfront).

Design for tomorrow's models

Tool scaffolding built for today's limitations becomes dead weight as models improve. The Vercel team's principle: "build for the model you expect to have, not the one you have today." Reasoning capabilities are improving faster than tool design patterns are updating.

More autonomy → more trust → better sandboxing

Giving agents full filesystem and terminal access is the right direction — but it requires shifting from permission prompts (which cause fatigue) to hard system-level boundaries that are always enforced regardless of what the agent decides.

Sandboxing: the security layer that enables autonomy

Claude Code's sandboxing reduced permission prompts by 84% in internal testing — not by granting more permissions, but by defining boundaries the agent can operate freely within. The two non-negotiable boundaries:

Filesystem isolation

Agent can only access specific directories. Built on Linux bubblewrap / macOS seatbelt. Prevents modification of system files, SSH keys, config outside the working directory — even if the agent is compromised via prompt injection.

Network isolation

All connections routed through a proxy enforcing domain allowlists. Blocks exfiltration of credentials, prevents malware downloads, stops unauthorised outbound API calls. Must be paired with filesystem isolation — either alone leaves a gap.

Sandbox technology landscape (2026)

Firecracker microVMs — strongest isolation

Hardware-enforced isolation with dedicated kernels per workload. ~125ms boot time, under 5 MiB overhead per instance. Used by AWS Lambda, E2B, Vercel Sandbox. Best for: untrusted code, multi-tenant agent platforms, regulated environments.

gVisor — syscall interception

Intercepts system calls before they reach the host kernel. Stronger than Docker, lighter than full VMs. 10–30% I/O overhead. Best for: compute-heavy agents where you control the code but want stronger-than-container isolation.

OS-level primitives (bubblewrap / seatbelt)

What Claude Code uses by default: lightweight, low-overhead, good for trusted in-house agents running known code on known machines. Provides meaningful protection against accidents without the operational complexity of VMs.

Managed platforms (E2B, Vercel Sandbox)

Production-ready microVM infrastructure with sub-second provisioning, pre-built agent runtimes, and billing per second. Right choice for building agent products that run user-provided or untrusted code at scale.

Four mandatory boundaries (converged guidance from Microsoft, NVIDIA, Anthropic)

①Network egress — block all outbound by default, allowlist only required endpoints. No DNS wildcards.
②Filesystem bounds — agent reads/writes only its working directory. System files, home directory, and env vars are outside scope.
③Secrets scoping — short-lived, task-scoped credentials only. Never pass long-lived API keys into agent context.
④Audit trail — immutable log of all agent actions. Permission denials are strong compromise indicators — alert on them.

11. The Lethal Trifecta: Agent Security

Simon Willison identified a critical security pattern: three capabilities, harmless individually, become a critical vulnerability when combined in one agent.

The Lethal Trifecta

🔒

Private data access

Emails, documents, databases, user records

🌐

Untrusted content

Web pages, emails, user uploads, external APIs

📤

External communication

HTTP requests, email sends, API calls, file writes

When all three exist: attackers embed instructions in untrusted content (a web page, an email) that trick the agent into exfiltrating private data through its external communication channel — without any user action.

Once an LLM agent has ingested untrusted input, it must be constrained so that it is impossible for that input to trigger any consequential actions. Everything eventually gets glued together into a sequence of tokens and fed to the model.

— Simon Willison, "The Lethal Trifecta for AI Agents"

Why AI-based defences don't work

Unlike parameterised SQL queries (which provide 100% SQL injection protection), confidence-based guardrails are non-deterministic. A 95% attack-blocking rate is a failing grade for security. You cannot defend against prompt injection with another LLM — the attack surface is the model itself.

Six Architectural Defences

① Action-Selector Pattern

Agents trigger tools but cannot see tool responses. Malicious data in tool output cannot influence subsequent behaviour.

② Plan-Then-Execute Pattern

Lock in all actions before any exposure to untrusted content. The plan cannot be modified after untrusted data is read.

③ LLM Map-Reduce Pattern

Isolated subagents operate on untrusted content independently. They aggregate results without cross-contamination.

④ Dual LLM Pattern

Privileged LLM coordinates a Quarantined LLM. The quarantined instance handles untrusted data and returns only symbolic variables ($VAR1) back — never raw data.

⑤ Code-Then-Execute Pattern

Privileged LLM generates code in a custom DSL enabling taint tracking. Untrusted data is tracked through every operation.

⑥ Context-Minimisation Pattern

Remove user prompts from context before returning results. Minimum information surface, minimum attack surface.

MCP creates the trifecta by design. MCP's architecture encourages combining private-data tools, untrusted-content tools, and external-communication tools in one agent. Survey data: 43% of tested MCP servers allowed command injection; 22% had arbitrary file read via path traversal; 30% were vulnerable to SSRF. Vet every MCP server before connecting. Never combine all three trifecta capabilities without architectural isolation.

11. Ten Lessons for Agentic Coding

From dbreunig.com's "10 Lessons for Agentic Coding" and the official Claude Code best practices. Hard-won guidance from teams shipping production code with agents.

① Implement to learn

Writing actual code surfaces overlooked decisions and improves specifications. The act of implementation teaches you what the spec missed.

② Rebuild often

Fork codebases, test radical ideas. Reconstruction is cheap; learning is valuable. Don't be precious about generated code.

③ Invest in end-to-end tests

Test behaviour, not implementation specifics. Behavioural contracts grant freedom to rebuild. Tests are the one thing that survives a rewrite.

④ Document intent, not code

Preserve the reasoning behind decisions, not the code itself. Code is cheap; understanding why it was written this way is not.

⑤ Keep specs in sync

Living documentation compounds learning. Update CLAUDE.md and specs continuously as decisions evolve. Stale specs mislead agents.

⑥ Find the hard stuff

Agents handle boilerplate. Humans tackle intuitive design, security, resilience, system architecture. Route your attention to what agents can't do reliably.

⑦ Automate everything easy

Convert learnings into reusable skills. Avoid getting trapped in endless manual re-prompting of the same workflows.

⑧ Develop your taste

When code generation outpaces your feedback mechanisms, personal judgment becomes critical. Agents produce working code; you produce good code.

⑨ Agents amplify experience

Technical expertise dramatically multiplies agent effectiveness. Experienced developers provide superior prompts, better constraints, better verification. Beginners benefit less.

⑩ Code is free as in puppies

Cheap to generate, expensive to maintain. Every line you accept becomes your responsibility. Prefer deletion to accumulation.

The Explore → Plan → Implement → Commit workflow

From Anthropic's official Claude Code best practices — the four-phase workflow that prevents solving the wrong problem at speed:

① Explore (plan mode on)

Claude reads files and answers questions without making changes. No code written yet. Use this to build a shared understanding of the codebase and the problem.

② Plan

Ask Claude to write a detailed implementation plan. Edit the plan directly before proceeding — this is your highest-leverage moment. A wrong plan executed fast is worse than no plan.

③ Implement (plan mode off)

Claude codes and verifies against its plan. Provide Claude with a way to verify its own work (tests, screenshots, API responses). This single investment yields the highest returns.

④ Commit

Ask Claude to commit with a descriptive message and open a PR. Atomic commits with clear messages make it easy to revert if something is wrong.

CLAUDE.md: what to include and exclude. Include: bash commands Claude can't guess, code style rules that differ from defaults, testing instructions, repo etiquette, architectural decisions. Do not include: things Claude can infer by reading code, standard conventions, detailed API docs (link instead), information that changes frequently. If Claude keeps violating a rule despite it being in CLAUDE.md, the file is probably too long and the rule is getting lost — prune ruthlessly.

13. HTML as Output — The Unreasonable Effectiveness

Thariq Shihipar (Claude Code team at Anthropic) wrote a widely-shared post arguing that HTML has replaced Markdown as the better default output format for AI agents. Simon Willison experimented with it and agreed. This is a simple, practical shift with outsized impact for demos and presentations.

The shift in one sentence: Markdown is limiting for files over ~100 lines and produces wall-of-text output. HTML lets Claude produce interactive, visually structured, shareable documents — slides, dashboards, annotated reviews, diagrams — with the same effort.

Presentations / slide decks

"Build a 12-slide HTML presentation on [topic] using scroll-snap sections, one slide per concept." Claude generates a complete deck you open in a browser and present immediately. No PowerPoint, no Keynote, no upload required. The track-c-intro.html on this site was built exactly this way.

PR / code reviews

"Help me review this PR by creating an HTML artifact with inline margin annotations and colour-coded findings by severity." The output is a navigable, scannable document rather than a wall of Markdown bullets.

Interactive explanations

"Explain this code in detail using HTML, CSS, and JavaScript to make the explanation rich, interactive, and as clear as possible." The result can include syntax highlighting, expandable sections, animated diagrams, step-by-step walkthroughs.

Dashboards and data reports

"Analyse this CSV and produce an HTML report with embedded charts." Claude writes the JavaScript inline. The file is self-contained — send it as an attachment, no server needed. Works for lab data, experiment results, literature surveys.

Why it works better than Markdown

SVG + CSS + JS inline

A single HTML file can contain diagrams (SVG), animations (CSS), interactivity (JS), and structured text — all in one self-contained document. Markdown can do none of these.

Shareable with zero infrastructure

One .html file. Open it in a browser, send it over email, drop it in OneDrive. No server, no login, no framework. The recipient needs only a browser tab.

Models are excellent at HTML

HTML is heavily represented in pretraining data — Claude is genuinely good at writing it. The reliability gap vs. Markdown has essentially disappeared with recent models.

Try it today

At the end of any Claude Code session where you've produced analysis or a report, add: Now produce the same content as a well-designed single-page HTML file with navigation and visual hierarchy. Make it self-contained and immediately openable in a browser.

Writing Effective Tools for Agents Sep 2025

tool design

More tools ≠ better outcomes. Namespace related tools. Return only high-signal information — replace cryptic UUIDs with semantic names. The "Think" Tool: Enabling Claude to Stop and Think Mar 2025

tool design

A simple tool that gives Claude a scratchpad to reason between tool calls — not extended thinking (before response), but mid-task reflection (after tool outputs). 54% relative improvement on the airline domain of τ-Bench. Use when: processing complex tool output, verifying policy compliance, sequential decision-making where each step builds on the last.

How We Built Our Multi-Agent Research System Jun 2025

architecture

Opus 4 lead + Sonnet 4 subagents outperformed single-agent Opus 4 by 90.2% on internal evaluations. Token usage explains 80% of performance variance on BrowseComp. Parallelisation reduced research time by up to 90%. Scaling rules: simple queries = 1 agent, complex research = 10+ agents with divided responsibilities.

Building a C Compiler with a Team of Parallel Claudes Feb 2026

case study

16 parallel Opus 4.6 instances in isolated Docker containers built a 100K-line Rust C compiler over 2,000 sessions. Key lesson: tests passing rarely means the job is done — the compiler still couldn't implement a 16-bit x86 code generator. GCC as "known-good oracle" enabled parallel agents on the same codebase. Lock-file synchronisation to prevent duplicate work.

Long-running agents and harnesses

Effective Harnesses for Long-Running Agents Nov 2025

architecture

Two-agent pattern (Initializer sets up the environment once; Coding agent makes incremental progress per session). Feature lists in JSON prevent premature completion. Progress files + git history replace in-context memory. Mandate single-feature focus per session — attempting too much at once is the primary failure mode.

Harness Design for Long-Running Application Development Mar 2026

architecture

GAN-inspired architecture: separate Generator and Evaluator agents. Models have "context anxiety" — they praise their own outputs even when quality is mediocre. Independent evaluators tuned to skepticism outperform self-evaluation. Grading criteria language directly shapes output ("museum quality" drove visual convergence). A 6-hour harness run ($200) vs solo 20-minute run ($9) showed dramatic quality differences.

Scaling Managed Agents: Decoupling Brain from Hands Apr 2026

architecture

As agents scale, decoupling planning ("brain") from execution ("hands") becomes essential. The brain sets strategy and evaluates progress; the hands carry out low-level tool calls. This separation prevents the planning context from being polluted by execution noise, and allows specialised models for each role.

Safety and practical ops

Claude Code Auto Mode: A Safer Way to Skip Permissions Mar 2026

safety

Auto mode automatically approves routine low-risk actions while asking for permission on high-risk ones. The alternative to --dangerously-skip-permissions for day-to-day use. Use --permission-mode auto to enable.

Demystifying Evals for AI Agents Jan 2026

evaluation

How to think about evaluating agent systems vs. one-shot models. The key insight: agent performance is path-dependent — the same final state can be reached in ways that differ greatly in reliability, cost, and safety. Evals must measure trajectories, not just outcomes.
